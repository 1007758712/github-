第一步：先将你本地的代码保存（Commit）

git add .
  
git commit -m "更新前端样式"

第二步：拉取云端最新代码（Pull）

git pull origin main

🚨 遭遇冲突（Conflict）：如果有其他人修改了和你相同文件的相同行
如果在第二步 Pull 的时候，因为别人修改的代码和你的代码撞车了，Git 就会报出**“合并冲突 (Merge Conflict)”**。这个时候千万别慌，不要直接强制推送，而是这样处理：

左侧面板会提示冲突： 在 VS Code 的“源代码管理”里，会多出一个分类叫 “合并更改 (Merge Changes)”，里面列出发生冲突的文件。
点开冲突文件进行修复： 点开这些文件，编辑器内部会用高亮标出冲突的地方，并浮现出几个方便的按钮：
采用当前更改 (Accept Current Change)：保留你刚刚写的代码，覆盖掉同事的。
采用传入的更改 (Accept Incoming Change)：保留同事在云端写的代码，覆盖掉你的。
保留双方更改 (Accept Both Changes)：把两边的代码都留下，然后你自己手动删改一下逻辑。
完成冲突修复后提交： 当所有冲突标注都被你点选修复后，保存文件。然后像平时提交一样，在这个文件旁边点击 + 号暂存，并点击提交 (Commit)。这个时候，你本地的代码就已经完美融合了你的工作和同事的工作。

第三步：推送到云端（Push）

git push origin branch-name



其中遭遇冲突cmd命令如下：

1. 查看哪些文件冲突了

git status

2. 手动打开并修改冲突文件
你会看到：<<<<<<< HEAD
这里是你本地刚写的代码
=======
这里是云端拉取下来的、别人写的代码
>>>>>>> 9a3b1e84d...

你的任务是： 删除掉那些多出来的符号行：<<<<<<< HEAD、======= 以及 >>>>>>> 瞎码。接着，决定保留上面那段、下面那段，还是把两段代码人工融合在一起。 最后保存文件并关闭。

3. 告诉 Git 冲突已经解决了
git add .

git commit -m "Resolve merge conflict"

git push origin branch-wzr


🚨 补充：如果冲突太乱搞砸了怎么办？ 如果你在改冲突时完全弄乱了文件，想退回到执行 git pull 之前的太平状态重新来过，可以在 CMD 敲入这行代码即可：

git merge --abort

>>使用pycharm操作指南：

编辑器的左侧面板点点点（最推荐、最简单）
第一步：暂存更改（Add / Stage）

点击左侧那个看起来像“分支”的源代码管理图标。
你会看到在“更改（Changes）”下面列出了你刚刚新增或修改的文件（现在应该是 4.28.md 和 interesting-project.md）。
将鼠标悬停在“更改（Changes）”这一栏的右侧，你会看到一个 ➕ 号（暂存所有更改 / Stage All Changes），点击它。
这一步的意思是：“Git，把这些文件的变动打包，准备发快递了。”
第二步：写提交信息（Commit）

在左侧面板顶部的输入框里，写下一句话，用来记录你这次修改了啥内容。比如你可以写：更新 4.28 热点项目。
写完后，点击输入框下面的 “提交（Commit）” 按钮。
这一步的意思是：“给快递贴上一个便签，这样以后翻历史记录就知道这次修改了什么。”
第三步：上传同步（Push）

提交完成后，你会看到刚才那个“提交”按钮变成了一个写着 “同步更改（Sync Changes）” 或者面板顶部有一个 “上传（Push）” 的按钮。
直接点击它！
这一步的意思是：“快递发车！把代码正式传到网上的仓库里。”
